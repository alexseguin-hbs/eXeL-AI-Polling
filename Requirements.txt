## 0) Product Summary

**Product name:** SoI Polling Tool (Polling Project)
**Purpose:** Enable fast, secure, large-group polling (100–10,000+ users) that turns free-form responses into **AI themes**, then into **ranked priorities**, then into **exportable reports**, while also supporting **SoI Trinity Tokens (♡ SI, 웃 HI, ◬ AI)** and a future-facing **cube simulation system** that enables humans (and eventually AI) to optimize modular code blocks safely.

**Core promise:** “Turn messy group input into clear priorities in under a minute (at scale), with governance + traceability + contribution value capture.”

## 1) Roles & Permissions

### 1.1 Roles

1. **Moderator** (runs the session)

* Creates session, configures mode (single-round vs multi-cycle), questions, anonymity rules.
* Opens/closes polls.
* Sees aggregated results.
* Can enable live prioritization (MVP2+).
* Can initiate deep dives (MVP3).
* Cannot silently change outcomes without audit (MVP3 governance).

2. **User (Participant)**

* Joins via QR/link.
* Submits responses (text MVP1, voice MVP2).
* Sees themes & ranks.
* Optionally sees personal vs group view (if enabled).
* Can flag token issues (MVP3+).

3. **Lead/Developer** (Team Leads and Developers are treated as one role label: **Lead/Developer**)

* Reviews prioritized outcomes and incorporates feedback.
* Can apply overrides with justification (MVP3 governance).
* Uses simulation mode to “check out” cube code, run tests, compare metrics, and propose improvements (MVP3+ simulation).

4. **Business Owner / Admin**

* Defines token rules/valuation defaults (per org).
* Manages paywall/Stripe unlock rules (optional).
* Owns compliance posture (RBAC, audit exports).

### 1.2 RBAC (minimum)

* **Moderator:** session config + control; view analytics for their sessions.
* **User:** submit + rank + view allowed outputs.
* **Lead/Developer:** audit view + override + simulation features (restricted).
* **Admin:** system-wide configuration + billing + token policy.

## 2) Workflow (User Journey) with Cubes

**Step 1.1 (Cube 1):** Moderator defines session objective, roles, questions, format.
**Step 1.2 (Cube 1):** System generates session ID + QR + share link.
**Step 2 (Cubes 2–3):** Users join and submit text/voice, anonymous or identified.
**Step 3 (Cubes 4–5):** Moderator opens/closes poll; System validates, timestamps, encrypts, stores.
**Step 4 (Cube 6):** System runs AI to generate themes, clusters, summaries, confidence.
**Step 5–6 (Cube 7):** Users rank; System aggregates into ordered priorities.
**Step 7 (Cubes 7–8):** Lead/Developer reviews outcomes, can override with justification, approves final decisions + token attribution.
**Step 8 (Cube 9):** System generates exports, reports, analytics, SoI token logs.
**Step 9 (Cube 10):** Simulation mode: Lead/Developer checks out a cube, improves code, runs replay tests, compares metrics, and proposes merge; prepares future AI self-optimization.

## 6) Token System Requirements (SoI Currency)

### 6.1 Token Definitions (must be visible in UI)

* **♡ Shared Intention (SI):** 1 minute of aligned contribution = **1 ♡**
* **웃 Human Intelligence (HI):** compensated skilled time; baseline conversion anchored to minimum wage (default $7.25/hr → ~7.25 웃/hr). Allow org override.
* **◬ Artificial Intelligence (AI):** time saved / scaled impact via automation; baseline multiplier example: **1 min SI = 5 ◬**

### 6.2 Token Ledger (minimum)

* Every token event must be logged:

  * session_id, user_id (or anonymous hash), timestamp, action_type, cube_id, delta_♡, delta_웃, delta_◬, rationale/inputs, version_id
* Token lifecycle states:

  * **simulated → pending → approved → finalized** (+ reversed if needed)
* Dispute workflow:

  * flagged_by, reason, evidence, status, resolution notes, resolved_by

### 6.3 Value Flow (MVP3+)

* Tokens can be:

  * redeemed (HI only; “treasury availability” config)
  * exchanged internally (services/time/tools/mentoring)
  * staked for governance (future)
  * amplified (SI converted to AI when automated outputs scale)

### 6.4 Business Clarity (CRS-35)

Per session, show:

* token rules and multipliers used
* definitions and examples
* what counts as contribution
* how overrides affect tokens

## 7) Simulation & Self-Optimization Requirements (Cube 10)

### 7.1 Simulation Concept (human-in-the-loop)

* Lead/Developer selects **Cube N** and clicks **“Check Out to Simulation”**
* System creates a sandbox environment:

  * pins dependencies (versions of adjacent cubes)
  * provides test fixtures + replay data (captured real sessions or synthetic load)
* Developer modifies code (in repo or in platform editor)
* Runs simulation:

  * unit tests for that cube
  * integration tests for upstream/downstream flows
  * load test (example: 1M inputs sampled) target: **< 60 seconds** for theme clustering pipeline (using sampling + batching + async jobs)
* Produces report:

  * latency, throughput, cost, error rate, memory, token correctness, audit integrity
* Lead/Developer submits “Check-In Proposal”
* Approver accepts/rejects; system versions and can roll back

### 7.2 Required Metrics

* latency per cube (p50/p95)
* throughput
* error rate
* cost estimate per 1k / 1M responses (AI + infra)
* correctness checks (deterministic rankings, token ledger integrity)
* dependency graph impact summary

### 7.3 Future AI Self-Optimization (not immediate build, but design for it)

* Keep simulation APIs consistent so “AI agent” can propose code diffs
* Humans remain final approvers (CRS-30)

## 11) Non-Functional Requirements

### Performance

* Join flow responsive on mobile
* AI processing must support scale:

  * **Goal example:** process **1M inputs via sampling/batching** and produce usable themes in **< 60 seconds** (simulation target), with stable degradation strategy (sampling, queueing, partial outputs).

### Security

* encryption at rest for responses
* RBAC enforcement across all endpoints
* audit logs for sensitive actions (override, token changes, exports)

### Privacy/Compliance

* anonymity mode = no PII stored
* data retention controls (later)
* export redaction (later)
* basic GDPR delete request flow (MVP2/3)

### Device & Responsive UI

* **Fully responsive design** — must work seamlessly on:
  * **Desktop** (1920px+)
  * **Laptop** (1024px–1440px)
  * **Tablet** (768px–1024px)
  * **Phone — Android** (360px–414px typical)
  * **Phone — iOS** (375px–430px typical)
* Mobile-first approach: all core flows (join, respond, rank, view themes) must be touch-friendly
* No native app required — responsive web UI via browser on all devices
* QR join flow must be optimized for mobile (primary entry point)
* Ranking UI must support both drag-drop (desktop/laptop) and tap-to-reorder (phone/tablet)
* Test across Chrome, Safari (iOS), Firefox, and Edge

### Accessibility

* WCAG-friendly UI
* keyboard navigation for ranking UI

### Observability

* logging, metrics dashboards, error tracking
* simulation vs production metric comparisons

## 12) Technology Decisions (Confirmed)

The following technology choices were confirmed during project setup:

### Frontend
- **Framework:** React + Next.js
- **UI Library:** shadcn/ui (clean, professional default)
- **Hosting:** Cloudflare Pages

### Backend
- **Framework:** FastAPI (Python) — as originally specified
- **Deployment:** VPS/cloud server with Cloudflare as CDN/reverse proxy
- **Rationale:** Python ecosystem required for AI/ML (embeddings, clustering, scikit-learn, etc.)

### Databases (Full Stack)
- **PostgreSQL:** Sessions, questions, rankings, audit logs, token ledger, users
- **MongoDB:** Raw responses (flexible schema for text/voice payloads)
- **Redis:** Real-time state — presence tracking, live rankings, WebSocket pub/sub, caching

### Authentication & RBAC
- **Provider:** Auth0
- **Roles:** Moderator, User (Participant), Lead/Developer, Business Owner/Admin

### AI Theme Clustering (Cube 6)
- **Architecture:** Multi-provider abstraction layer (provider-agnostic interface)
- **Initial providers:** Implement as needed (OpenAI, Anthropic, Cohere, open-source)
- **Pipeline:** Embeddings → Clustering → Summarization

### Scale
- **MVP1 target:** 2,000–10,000 concurrent users per session
- **AI simulation target:** 1M inputs via sampling/batching < 60 seconds

### Build Approach
- **Scaffold all Cubes 1-9,** then implement cube by cube
- **Cube 5 (Gateway/Orchestrator) is built FIRST** — center of the 3x3 grid
- **Implementation order:** 5 → 1 → 2 → 4 → 6 → 7 → 9
- **Deferred:** Cube 3 (MVP2), Cube 8 (MVP3), Cube 10 (MVP3, paused)

### Time Tracking (Built into Cube 5 from Day 1)
- Track **active participation time** per user per session
- Start timer when user begins responding or voting; stop on submit/complete
- Per-action timestamps: start_time, end_time for each response and ranking action
- **1 minute active participation = 1 ♡ SI token**
- **Only SI tokens are tracked during polling/voting** — HI tokens (웃) are for later project execution phases, not polling

### Monetization (MVP1 — Stripe)
- **Free tier:** Small sessions at no cost
- **Moderator pays:** Per-session fee for larger sessions
- **User pays for results:** Users pay to download full results/reports
- **Lead/Developer exception:** Leads ALWAYS get free access to results (transparency + accountability)
- Stripe integration set up from MVP1
