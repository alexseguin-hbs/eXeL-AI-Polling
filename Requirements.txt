## 0) Product Summary

**Product name:** SoI Governance Engine (eXeL-AI-Polling)
**Purpose:** Production-scale, deterministic, horizontally scalable Governance Compression Infrastructure that turns free-form responses into **AI themes**, then into **ranked priorities**, then into **exportable reports**, while supporting **SoI Trinity Tokens (♡ SI, 웃 HI, ◬ AI)**, governance compression, and a cube simulation system that enables humans (and eventually AI) to optimize modular code blocks safely.

**Core promise:** "Turn messy group input into clear priorities in under a minute (at scale), with governance + traceability + contribution value capture."

**Delivery model:** API-first platform with SDK. The entire tool is designed to be embedded into external products, websites, and codebases via API calls and client SDKs — not just a standalone web app. Think embeddable governance infrastructure (like an iframe, but fully modular and API-driven at scale).

**Scale targets:**
* **Stable:** 100,000 concurrent users
* **Burst:** 1,000,000 concurrent capacity
* **AI processing:** 1M inputs via sampling/batching in < 60 seconds
* **Theme determinism:** Identical inputs MUST yield identical themes

#### Pixelated Tokens (Image Export Format)

A self-contained, value-carrying image that encodes a user's earned tokens. The image is structured as a bordered frame around a center QR code:

* **Top pixel line:** Row of colored pixels — color and position of each pixel encodes a character. Encoded data includes: Session Name, Session ID, Date, Time, ♡ (SI) earned, ◬ (AI) earned, 웃 (HI) earned, User hash, Project ID, encoding version
* **Bottom pixel line:** **Mirror/reverse of top line** — DNA-style integrity check. Top read forward and bottom read backward must match to confirm authenticity. Mismatch = tampered/invalid
* **Left vertical pixel line:** Encryption key (first half) — required to decode the top/bottom pixel data
* **Right vertical pixel line:** Encryption key (second half) — combined with left vertical to form the full decryption key
* **Center:** QR code — scannable for quick verification and token data retrieval
* **Pixel encoding scheme:** Versioned color-to-character mapping. Each pixel = one encoded character. Color + position = value. Left + right verticals provide the key to translate.

#### Delivery & Data Destruction

* **Delivery methods (user chooses one):**
  * Download directly from app
  * System sends via text/SMS
  * System sends via email
* **Data destruction:** After the Pixelated Token image is delivered to the user, **the user's token data is destroyed from the system**. The token value lives in the image itself, not on the server. Privacy-by-design — the system does not retain user token data post-delivery.
* **Implication:** Users are responsible for safekeeping their Pixelated Token image. Lost image = lost token proof.

* **Methods 2 & 3 export:** Full content package per session attached to Project ID — includes meeting purpose, Desired Outcome, time logged, participant roles, results, outcome status, ♡ earned. All participants automatically receive export on completion.

## 6) Token System Requirements (SoI Currency)

### 6.1 Token Definitions (must be visible in UI)

**Symbols are the primary identifiers** — no need to write "SI", "HI", "AI" in UI/tracking:

* **♡:** 1 minute of aligned contribution = **1 ♡** (Shared Intention) — distributed via **3 methods at launch** (see 6.5)
* **웃:** compensated skilled time; baseline anchored to jurisdiction minimum wage. Default $7.25/hr (Austin, Texas / US federal). **59 jurisdictions loaded** — 9 international ($0.34–$3.02/hr) + 50 US states ($7.25–$16.28/hr). Currently `hi_enabled=False` (pre-treasury); flip to True when funded.
* **◬:** time saved / scaled impact via automation; default multiplier: **◬ = 5x ♡**

**웃 vision:** Pay out globally at local minimum wage to leverage global talent. Rate resolved per participant from jurisdiction rate table.

**Token independence:** ♡ (SI), ◬ (AI), and 웃 (HI) track value independently with no dependency on Quai, Qi, or any cryptocurrency/blockchain integration. The token system is fully functional as a standalone value-tracking layer. Crypto/blockchain integrations (Quai, Qi, or others) can be introduced later as optional layers when available.

### 6.2 ♡ SI Distribution Methods (3 Methods at Launch)

♡ tokens are earned via time-based contribution. At launch, the system supports **3 distinct distribution methods:**

#### Method 1 — Polling Contribution
* The existing model: 1 minute of active participation in a polling session = 1 ♡
* Tracked via Cube 5 time tracking (login, responding, ranking, reviewing)
* All participants earn ♡ for their active time

#### Method 2 — Peer-to-Peer Volunteer (2 People + 1 Witness)
* One user volunteers to help another user's project — **no monetary payment, only ♡ SI**
* **Minimum 3 users present:** Helper, Recipient, and Witness (witness can be any registered user)
* **Before starting:**
  * All 3 users must be present
  * The group documents the **Desired Outcome** in the system
  * Timer cannot start until Desired Outcome is logged and all 3 confirm
* **During task:** Timer runs via Cube 5 time tracking; all participants earn ♡ for time present
* **At stop of task:**
  * All 3 users assess whether the Desired Outcome was achieved
  * Outcome status logged: **achieved / partially achieved / not achieved**
  * ♡ tokens are awarded for time contributed regardless of outcome (you showed up and worked)
  * Outcome result is permanently attached to the record for accountability and reputation
* All content and results are exportable and attached to the Project ID

#### Method 3 — Team Collaboration (3-Person Minimum, Role-Required)
* A team of **3 or more** works on a project together
* **Required role coverage** (minimum 1 of each, can have more members):
  * **Technology Representative** — technical execution and feasibility
  * **Creative Representative** — design, UX, ideation, content
  * **Business / Value Representative** — strategy, ROI, market fit, stakeholder value
* **Before starting:**
  * All team members agree on: **meeting purpose, time estimate, and Desired Outcome**
  * Agreement is logged in the system; timer cannot start until all members confirm
* **During task:** Timer runs via Cube 5 time tracking; all participants earn ♡ for time present
* **After meeting:**
  * Team logs results against the original Desired Outcome
  * All members confirm/sign off on the results
  * Outcome status logged: **achieved / partially achieved / not achieved**
  * ♡ tokens awarded for time contributed regardless of outcome
* **Export:** All users receive an export of the full content (meeting purpose, desired outcome, time logged, results, ♡ earned) attached to the **Project ID**

### 6.3 Token Ledger (minimum)

* Every token event must be logged (append-only ledger):

  * session_id, user_id (or anonymous hash), timestamp, action_type, cube_id, ♡, 웃, ◬, rationale/inputs, version_id
  * **distribution_method:** `polling` | `peer_volunteer` | `team_collaboration` — identifies which of the 3 methods generated the entry
  * **desired_outcome_id:** Reference to the Desired Outcome record (Methods 2 & 3)
  * **outcome_status:** `achieved` | `partially_achieved` | `not_achieved` | `n/a` (Method 1)
* Token lifecycle states:

  * **simulated → pending → approved → finalized** (+ reversed if needed)
* Dispute workflow:

  * flagged_by, reason, evidence, status, resolution notes, resolved_by

### 6.4 Value Flow (MVP3+)

* Tokens can be:

  * redeemed (HI only; “treasury availability” config)
  * exchanged internally (services/time/tools/mentoring)
  * staked for governance (future)
  * amplified (SI converted to AI when automated outputs scale)

## 8.5) API-First Platform & SDK (Embeddable Integration)

### 8.5.1 Architecture Principle

The entire SoI Governance Engine is **API-first**. Every feature accessible in the web UI must also be accessible via a public REST API. This enables external companies to run the governance engine inside their own code, websites, or products — without ever visiting the SoI web app.

### 8.5.2 SDK (Client Libraries)

* Provide official SDKs (JavaScript/TypeScript first, Python second) for embedding
* SDK wraps all REST API endpoints with typed methods
* SDK handles: authentication (API key + OAuth), session management, webhook callbacks, error handling
* Embeddable UI components (Web Components / iframe-based) for companies that want drop-in UI
* Headless mode: companies can use the API/SDK with their own custom UI

### 8.5.3 Scoping Hierarchy: Project → Differentiator → Specification

Every SDK/API interaction is scoped to a three-level hierarchy:

* **Project (Project ID):** Top-level container representing a company's product, initiative, or system. Each project has its own configuration, sessions, token rules, and data isolation.
* **Differentiator (Differentiator ID):** Within a project, differentiators represent distinct dimensions, features, or hypotheses being evaluated. Example: "UX Approach A vs B", "Pricing Model X", "Feature Priority Set".
* **Specification (Specification ID):** Within a differentiator, specifications are the concrete parameters, constraints, or configurations to fine-tune and simulate. Example: target audience, pricing tiers, feature toggles, governance rules.

This hierarchy allows companies to:
* Run multiple polling/governance sessions scoped to specific projects
* A/B test ideas via differentiators
* Fine-tune and simulate existing products or new concepts via specifications
* Compare outcomes across differentiators and specifications within a project

### 8.5.4 API Key & Tenant Management

* API keys issued per organization (tenant)
* Each API key scoped to one or more projects
* Rate limiting per API key (separate from web UI rate limits)
* Usage metering for billing (API calls, sessions, AI processing)
* Webhook registration for async events (themes ready, ranking complete, etc.)

### 8.5.5 Embeddable Modes

| Mode | Description |
|------|-------------|
| **Full embed** | Drop-in UI component (iframe / Web Component) — full session flow rendered inside host app |
| **Headless API** | No UI — company calls API/SDK endpoints and renders their own UI |
| **Hybrid** | Company uses some embedded components (e.g., ranking widget) and custom UI for others |

### 8.5.6 Core Entities (additions for SDK scoping)

**Project**
* id, org_id, name, description, created_at, config (token rules, AI provider, language defaults)

**Differentiator**
* id, project_id, name, description, hypothesis, created_at

**Specification**
* id, differentiator_id, parameters (JSON), constraints, version, created_at

**Session** (updated)
* Now includes: project_id, differentiator_id, specification_id (all optional for standalone web use; required for SDK/API use)

### Monetization (MVP1 — Stripe)

#### Moderator Fee (Required)
- **Moderator must pay** to use the tool (per-session fee via Stripe)
- **Cost splitting option:** Moderator can choose to split the fee equally among participants
  * Example: $100 fee / 100 users = $1 per person
  * Example: $100 fee / 200 users = $0.50 per person
  * Fee per user is calculated dynamically as participants join (displayed in real-time)
  * If Moderator does not enable cost splitting, Moderator pays the full fee
- **Paying members receive Polling Results** — any user who pays their share automatically receives the full session output

#### Gamified Contribution Reward
- **Moderator-funded bonus:** Moderator can set a reward amount (e.g., $25) to be awarded to one participant
- **Appears random to users** but is determined by the **Contribution Quality Score (CQS)** — a hidden rating system
- **Winner criteria:** The participant whose response falls within the **#1 most-voted Theme (Theme2 level)** and has the highest CQS
- **CQS Rating Metrics (hidden from users, visible to Moderators and system):**
  * **Insight Score (0–100):** Does the response reveal a non-obvious perspective, connection, or pattern? AI-assessed against the full response corpus
  * **Depth Score (0–100):** Does the response go beyond surface-level commentary? Measures specificity, examples, supporting reasoning, and detail relative to session average response length
  * **Future Impact Score (0–100):** Does the response address forward-looking implications, emerging trends, risks, or opportunities? AI-assessed for temporal orientation and strategic relevance
  * **Originality Score (0–100):** How unique is this response compared to others in the same theme cluster? Measured via embedding distance from cluster centroid — further (while still in-cluster) = more original
  * **Actionability Score (0–100):** Does the response suggest concrete next steps, solutions, or implementable ideas?
  * **Relevance Score (0–100):** How closely does the response align with the question asked and the theme it was clustered into? Embedding similarity to question + theme centroid
- **CQS Composite:** Weighted average of all 6 metrics. Default weights: Insight 20%, Depth 15%, Future Impact 25%, Originality 15%, Actionability 15%, Relevance 10%. Moderator can adjust weights per session (advanced config).
- **Visibility:** CQS scores hidden from all users. Moderators see scores in their dashboard. System uses scores for reward selection and analytics.
- **Tie-breaking:** If CQS tie, earlier submission wins (first-mover advantage)

#### Access Rules
- **Lead/Developer exception:** Leads ALWAYS get free access to results (transparency + accountability)
- Stripe integration set up from MVP1

## 13) Production Architecture Constraints

### Architectural Rules
* **No row-by-row API calls** — batch all embedding requests efficiently
* **No disk-heavy CSV intermediates** — stream data through pipeline in memory
* **No repeated full clustering** — use streaming/incremental MiniBatchKMeans updates
* **Horizontal scaling** — all services must be stateless or use shared state (Redis/Postgres)
* **Rate limiting** on all public endpoints
* **Anti-sybil safeguards** on voting and response submission
* **Governance weight damping** to prevent manipulation
* **Circuit breaker pattern** for all external AI provider calls (OpenAI, Grok, Gemini)
* **Graceful degradation** — system must serve partial results if AI pipeline is delayed

### Determinism Requirements
All clustering and ranking operations must be fully reproducible:
* **Seeded MiniBatchKMeans** — fixed `random_state` for identical outputs
* **Fixed embedding model version** — pin exact model ID, never auto-upgrade
* **Stable cluster ordering** — deterministic sort after clustering
* **Replay reproducibility hash** — SHA-256 of (inputs + parameters) = expected output hash
* **Version-locked cube dependency graph** — every cube pins its upstream/downstream versions

### Background Processing Architecture
* **Task queue:** Celery or async worker pool for background AI processing
* **Embedding worker fleet:** Dedicated batch workers, horizontally scaled
* **Streaming clusterer:** MiniBatchKMeans with deterministic seed, incremental updates
* **Governance engine:** Voting weight compression, quadratic normalization
* **Append-only token ledger:** Immutable entries with lifecycle states

### Failure Mode Handling
* **AI provider outage:** Circuit breaker → queue → exponential backoff → failover to next provider → cached embedding fallback
* **Embedding backlog surge:** Back-pressure → sampling mode → priority queue
* **Partial cluster update:** Atomic batch commits → rollback on failure → serve last stable cluster
* **Redis failure:** Graceful degradation to Postgres → reconnect with backoff
* **Postgres failover:** Read replicas → automatic promotion → connection pool retry
* **Burst queue overflow:** Load shedding via sampling → 503 + retry-after header
* **Governance manipulation:** Anti-sybil detection → weight damping → anomaly flagging → audit log

## 14) Governance Compression & Token Math

See `Token_Governance_Math.md` for full formal specification. Summary:

### Governance Requirements
* **Governance weight damping** — prevent any single actor from dominating outcomes
* **Quadratic vote normalization** — diminishing returns on repeated votes: `weight = sqrt(tokens_staked)`
* **Token velocity caps** — limit transfer/redemption speed to prevent gaming
* **Reputation multipliers** — earned trust increases governance weight over time
* **Drift detection** — `cosine(previous_centroid, new_centroid)` to detect theme drift between runs
* **Cluster stability scoring** — measure consistency of theme assignments across re-runs

### Treasury Accounting
* **Treasury-backed HI redemption** — HI tokens only redeemable against available treasury balance
* **Token dispute workflow** — flag → review → resolve with complete audit trail
* **Version-locked ledger** — every entry references cube version + dependency graph hash
