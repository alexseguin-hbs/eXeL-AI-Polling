## 0) Product Summary

**Product name:** SoI Governance Engine (eXeL-AI-Polling)
**Purpose:** Production-scale, deterministic, horizontally scalable Governance Compression Infrastructure that turns free-form responses into **AI themes**, then into **ranked priorities**, then into **exportable reports**, while supporting **SoI Trinity Tokens (♡ SI, 웃 HI, ◬ AI)**, governance compression, and a cube simulation system that enables humans (and eventually AI) to optimize modular code blocks safely.

**Core promise:** "Turn messy group input into clear priorities in under a minute (at scale), with governance + traceability + contribution value capture."

**Delivery model:** API-first platform with SDK. The entire tool is designed to be embedded into external products, websites, and codebases via API calls and client SDKs — not just a standalone web app. Think embeddable governance infrastructure (like an iframe, but fully modular and API-driven at scale).

**Scale targets:**
* **Stable:** 100,000 concurrent users
* **Burst:** 1,000,000 concurrent capacity
* **AI processing:** 1M inputs via sampling/batching in < 60 seconds
* **Theme determinism:** Identical inputs MUST yield identical themes

## 1) Roles & Permissions

### 1.1 Roles

1. **Moderator** (runs the session)

* Creates session, configures mode (single-round vs multi-cycle), questions, anonymity rules.
* Opens/closes polls.
* Sees aggregated results.
* Can enable live prioritization (MVP2+).
* Can initiate deep dives (MVP3).
* Cannot silently change outcomes without audit (MVP3 governance).

2. **User (Participant)**

* Joins via QR/link.
* Submits responses (text MVP1, voice MVP2).
* Sees themes & ranks.
* Optionally sees personal vs group view (if enabled).
* Can flag token issues (MVP3+).

3. **Lead/Developer** (Team Leads and Developers are treated as one role label: **Lead/Developer**)

* Reviews prioritized outcomes and incorporates feedback.
* Can apply overrides with justification (MVP3 governance).
* Uses simulation mode to “check out” cube code, run tests, compare metrics, and propose improvements (MVP3+ simulation).

4. **Business Owner / Admin**

* Defines token rules/valuation defaults (per org).
* Manages paywall/Stripe unlock rules (optional).
* Owns compliance posture (RBAC, audit exports).

### 1.2 RBAC (minimum)

* **Moderator:** session config + control; view analytics for their sessions.
* **User:** submit + rank + view allowed outputs.
* **Lead/Developer:** audit view + override + simulation features (restricted).
* **Admin:** system-wide configuration + billing + token policy.

## 2) Workflow (User Journey) with Cubes

**Step 1.1 (Cube 1):** Moderator defines session objective, roles, questions, format.
**Step 1.2 (Cube 1):** System generates session ID + QR + share link.
**Step 2 (Cubes 2–3):** Users join and submit text/voice, anonymous or identified.
**Step 3 (Cubes 4–5):** Moderator opens/closes poll; System validates, timestamps, encrypts, stores.
**Step 4 (Cube 6):** System runs AI to generate themes, clusters, summaries, confidence.
**Step 5–6 (Cube 7):** Users rank; System aggregates into ordered priorities.
**Step 7 (Cubes 7–8):** Lead/Developer reviews outcomes, can override with justification, approves final decisions + token attribution.
**Step 8 (Cube 9):** System generates exports, reports, analytics, SoI token logs.
**Step 9 (Cube 10):** Simulation mode: Lead/Developer checks out a cube, improves code, runs replay tests, compares metrics, and proposes merge; prepares future AI self-optimization.

## 3) Cube Architecture (Modules)

### Cube 1 — Session Join & QR (1,2,2) — CENTER of Level 1

> **Implementation Status:** IMPLEMENTED (~70% of spec). CRS-01→CRS-06 complete. Core CRUD, state machine, QR, join, presence, questions, capacity enforcement, CQS weights, 11 new session model fields, 3 new participant fields, Token HUD, Timer Context, Voice Input stub, Cube Architecture Status Panel, one-question-at-a-time UX all implemented. Remaining: Stripe payment flow, cost splitting logic, scoping context, master language backend tables, desired outcomes (M2/M3), multi-device WebSocket sync, live feed, metrics. See CLAUDE.md § "Cube 1" for full breakdown + test procedure.

* **All code is modular** — every cube is a self-contained module with clean interfaces; all aspects of each cube must be modular and independently maintainable
* Session create — supports 3 session types: **polling** (Method 1), **peer_volunteer** (Method 2), **team_collaboration** (Method 3)
* **Moderator session config (at creation):**
  * **Polling mode:** Single Round | Multi-Round Deep Dive
  * **Scoping context:** Moderator selects one of Project, Specification, or Product Differentiator and links to existing scoping ID
  * **Capacity / Pricing tier:**
    * **Free:** ≤19 participants — no payment required
    * **Moderator Paid:** Moderator pays full session fee (Stripe/GPay/ApplePay) — no user payment
    * **Moderator+User Cost Split:** Moderator sets base fee; fee is divided equally among participants dynamically. All paying participants receive results.
  * **Max participants:** Configurable per session (default unlimited for paid tiers, hard cap 19 for Free tier)
  * **Session fee:** Amount (required for paid tiers, via Stripe/GPay/ApplePay)
  * **Gamified reward:** Enable/disable — if enabled, set bonus amount (e.g., $25) and optionally adjust CQS metric weights
  * **Ranking mode:** `auto` (system aggregates) / `manual` (Moderator curates) / `live` (real-time WebSocket updates)
  * **Theme2 voting level:** Moderator selects `theme2_9` (9 themes), `theme2_6` (6 themes), or `theme2_3` (3 themes) — determines granularity of themes users vote on. Can be set at creation or after Theme 1 results are shown.
  * **Live response feed:** Enable/disable — if enabled, hosting PC displays scrolling 33-word summaries of responses in real-time (paid tiers only)
  * **Response limits:** Max response character limit (default 500), max question character limit (default 500)
* **Moderator multi-device support:** Moderator can manage a session from **PC and Phone simultaneously**. Session control state is synced in real-time via WebSocket/Redis. Phone-optimized Moderator UI shows: session status, participant count, open/close poll controls, quick stats. PC UI shows: full dashboard, analytics, configuration, QR management. Both devices share the same session auth token with device-aware layout adaptation.
* **Polling modes:**
  * **Single Round:** One question cycle → responses → AI themes → ranking → results. No follow-up.
  * **Multi-Round Deep Dive:** Iterative cycles (up to 3 configurable). After initial results, Moderator selects top themes → system generates follow-up questions → participants respond again → re-cluster → re-rank. Preserves context across rounds.
* Session ID generation
* QR + share link generation
* **User join flow (sequential screens after QR scan):**
  1. **Language selection gate:** Dropdown with all active languages (33 at launch, extensible via Master language table)
  2. **Results opt-in + payment (click required):** "Would you like to receive a copy of the polling results?" — must actively click Yes or No. If Moderator enabled cost splitting, the per-user fee is displayed (e.g., "$0.50 for results"). Payment processed via Stripe, Google Pay, or Apple Pay if Yes. Paying users flagged for Cube 9 results distribution. (Free tier: no payment screen.)
  3. **Session join** → see question(s)
  * *(Note: Scoping context is set by Moderator at session creation — users do NOT select scoping in the join flow.)*
* **Master UI/UX language table:** A centralized, extensible language registry that drives all UI translations and language options. Admins and developers can add new languages to expand the tool without code changes. All cubes reference this single table for language support.
* Join flow + presence setup
* **Method 2/3 setup flow:** Desired Outcome input, role assignment (Method 3: Tech/Creative/Business), time estimate, all-participant confirmation gate before timer starts
* Session state: draft/open/closed/archived
* Basic security checks

#### Cube 1 Traceability
* Every session is linked to exactly ONE scoping context **set by the Moderator at session creation**: **Project ID**, **Specification ID**, or **Product Differentiator ID**
* `sessions.scoping_type` + `sessions.scoping_id` provide the link
* All downstream data (responses, themes, rankings, tokens, exports) inherits scoping context via `session_id` → `sessions.scoping_type/scoping_id`
* This enables filtering, aggregation, and feedback loops per Project, Specification, or Differentiator

### Cube 2 — Text Submission Handler (1,2,3)

* Validate text inputs **in all active languages** (references Master UI/UX language table from Cube 1)
* Enforce limits: **question ≤ 500 chars**; response length configurable (default ≤ 500) — character limits must account for multi-byte Unicode (CJK, Arabic, etc.)
* Language-aware input: user submits in their selected language; language tag stored with each response for downstream AI processing (Cube 6)
* **Immediate token display:** After text submission, user sees their earned ♡ (SI) and ◬ (AI) tokens on screen (◬ = 5x ♡ minutes). Calculated by Cube 5, displayed inline.
* Optional anonymization
* Basic profanity/PII detection (optional) — must support multilingual detection across all active languages
* **Live 33-Word Response Feed (Hosting PC):**
  * As users submit text (or voice-to-text from Cube 3), the **Moderator's hosting PC** displays a **scrolling live feed of 33-word summaries** of each response. Summaries appear in real-time as inputs arrive, scrolling down while others are still finishing.
  * **Toggleable:** Moderator can enable/disable the live feed at session creation or during the session.
  * **Paid-tier feature:** The live feed may be restricted to **paid tiers only** (Moderator+User Cost Split). Free-tier sessions do not show the live feed.
  * **Purpose:** Encourages user engagement — participants see summarized responses appearing in real-time on the projected/shared screen.
  * **33-word summaries** are generated by Cube 6 AI pipeline in near-real-time (streamed as each response is submitted, before full theme clustering).
  * **Privacy:** Summaries shown are anonymized (no user attribution on the feed). Profanity-flagged responses are excluded from the feed.

### Cube 5 — User Input Gateway / Orchestrator (1,3,1)

* Central gateway for: join events, submissions, poll state updates
* Triggers AI processing after poll close
* Triggers ranking aggregation after ranking close
* **Time tracking for all 3 ♡ methods:**
  * Method 1: Track active polling participation (login, responding, ranking, reviewing)
  * Method 2: Track peer volunteer session — timer starts after all 3 confirm, stops when task ends
  * Method 3: Track team collaboration session — timer starts after all members confirm, stops when meeting ends
* **Confirmation gates:** Timer cannot start until all required participants confirm presence + Desired Outcome (Methods 2 & 3)
* **Post-task flow:** Trigger outcome assessment (all participants sign off), log results, calculate ♡ for all participants
* **Immediate token display:** On every submission (text/voice), Cube 5 calculates current ♡ and ◬ (= 5x ♡ minutes) and returns them to the frontend for instant display

* **Payment orchestration:** Coordinates Stripe payment flow during join (Cube 1 → Stripe → Cube 4 payment status) and reward disbursement after session close

### Cube 6 — AI Theming Clusterer (1,2,1)

* Embeddings + clustering + summarization
* Returns: themes, subthemes, confidence, counts, exemplars
* Must support "Theme-01 basic" first; "Theme-02 enhanced" later; deep dives in MVP3
* **Live 33-word summary generation:** As responses arrive (during input phase), Cube 6 generates 33-word summaries in near-real-time. These feed into the **Live Response Feed** on the Moderator's hosting PC (Cube 2 feature). Summaries are streamed as each response is submitted, before full theme clustering begins.
* **Theme Generation Flow (triggered when Moderator locks input):**
  1. **Moderator locks voting/input** → triggers Theme 1 generation
  2. **Theme 1 (Risk/Supporting/Neutral):** All responses classified into 3 primary categories with confidence scores
  3. **Random sampling of 10:** For Theme 2 sub-theme generation, the system randomly samples 10 responses to seed the 3-theme clustering (performance optimization for large datasets)
  4. **Theme 2 progressive generation (paid tiers):** For paid sessions (Moderator Paid / Cost Split), Theme 2 results appear on the **hosting PC screen progressively** — themes display one by one in random order as the AI completes each assignment, creating engagement while processing continues
  5. **Theme 2 completion:** Once all Theme 2 assignments are done, the full set is presented for voting
* **Moderator-selected Theme2 level for voting:** Moderator configures which Theme2 granularity level users vote on:
  * **Theme2_9:** Full 9-theme set (most granular)
  * **Theme2_6:** Reduced 6-theme set (medium granularity)
  * **Theme2_3:** Reduced 3-theme set (broadest)
  * Moderator chooses this at session creation or after Theme 1 results are shown
* **Contribution Quality Score (CQS) engine:** After clustering and ranking (Cube 7), Cube 6 scores ONLY eligible responses — those in the #1 most-voted Theme2 cluster with >95% theme confidence — using the 6 CQS metrics:
  * **Insight Score (0–100):** AI-assessed — non-obvious perspectives, connections, patterns against full response corpus
  * **Depth Score (0–100):** Specificity, examples, reasoning; detail relative to session average response length
  * **Future Impact Score (0–100):** AI-assessed — forward-looking implications, trends, risks, strategic relevance
  * **Originality Score (0–100):** Embedding distance from cluster centroid — further (while in-cluster) = more original
  * **Actionability Score (0–100):** AI-assessed — concrete next steps, solutions, implementable ideas
  * **Relevance Score (0–100):** Embedding similarity to question + theme centroid
  * **CQS Composite:** Weighted average. Default weights: Insight 20%, Depth 15%, Future Impact 25%, Originality 15%, Actionability 15%, Relevance 10%. Moderator-adjustable per session.
* CQS scores stored per response; hidden from users, visible to Moderators and system

#### Pixelated Tokens (Image Export Format)

A self-contained, value-carrying image that encodes a user's earned tokens. The image is structured as a bordered frame around a center QR code:

* **Top pixel line:** Row of colored pixels — color and position of each pixel encodes a character. Encoded data includes: Session Name, Session ID, Date, Time, ♡ (SI) earned, ◬ (AI) earned, 웃 (HI) earned, User hash, Project ID, encoding version
* **Bottom pixel line:** **Mirror/reverse of top line** — DNA-style integrity check. Top read forward and bottom read backward must match to confirm authenticity. Mismatch = tampered/invalid
* **Left vertical pixel line:** Encryption key (first half) — required to decode the top/bottom pixel data
* **Right vertical pixel line:** Encryption key (second half) — combined with left vertical to form the full decryption key
* **Center:** QR code — scannable for quick verification and token data retrieval
* **Pixel encoding scheme:** Versioned color-to-character mapping. Each pixel = one encoded character. Color + position = value. Left + right verticals provide the key to translate.

#### Delivery & Data Destruction

* **Delivery methods (user chooses one):**
  * Download directly from app
  * System sends via text/SMS
  * System sends via email
* **Data destruction:** After the Pixelated Token image is delivered to the user, **the user's token data is destroyed from the system**. The token value lives in the image itself, not on the server. Privacy-by-design — the system does not retain user token data post-delivery.
* **Implication:** Users are responsible for safekeeping their Pixelated Token image. Lost image = lost token proof.

* **Methods 2 & 3 export:** Full content package per session attached to Project ID — includes meeting purpose, Desired Outcome, time logged, participant roles, results, outcome status, ♡ earned. All participants automatically receive export on completion.

## 6) Token System Requirements (SoI Currency)

### 6.1 Token Definitions (must be visible in UI)

**Symbols are the primary identifiers** — no need to write "SI", "HI", "AI" in UI/tracking:

* **♡:** 1 minute of aligned contribution = **1 ♡** (Shared Intention) — distributed via **3 methods at launch** (see 6.5)
* **웃:** compensated skilled time; baseline anchored to jurisdiction minimum wage. Default $7.25/hr (Austin, Texas / US federal). **59 jurisdictions loaded** — 9 international ($0.34–$3.02/hr) + 50 US states ($7.25–$16.28/hr). Currently `hi_enabled=False` (pre-treasury); flip to True when funded.
* **◬:** time saved / scaled impact via automation; default multiplier: **◬ = 5x ♡**

**웃 vision:** Pay out globally at local minimum wage to leverage global talent. Rate resolved per participant from jurisdiction rate table.

**Token independence:** ♡ (SI), ◬ (AI), and 웃 (HI) track value independently with no dependency on Quai, Qi, or any cryptocurrency/blockchain integration. The token system is fully functional as a standalone value-tracking layer. Crypto/blockchain integrations (Quai, Qi, or others) can be introduced later as optional layers when available.

### 6.2 ♡ SI Distribution Methods (3 Methods at Launch)

♡ tokens are earned via time-based contribution. At launch, the system supports **3 distinct distribution methods:**

#### Method 1 — Polling Contribution
* The existing model: 1 minute of active participation in a polling session = 1 ♡
* Tracked via Cube 5 time tracking (login, responding, ranking, reviewing)
* All participants earn ♡ for their active time

#### Method 2 — Peer-to-Peer Volunteer (2 People + 1 Witness)
* One user volunteers to help another user's project — **no monetary payment, only ♡ SI**
* **Minimum 3 users present:** Helper, Recipient, and Witness (witness can be any registered user)
* **Before starting:**
  * All 3 users must be present
  * The group documents the **Desired Outcome** in the system
  * Timer cannot start until Desired Outcome is logged and all 3 confirm
* **During task:** Timer runs via Cube 5 time tracking; all participants earn ♡ for time present
* **At stop of task:**
  * All 3 users assess whether the Desired Outcome was achieved
  * Outcome status logged: **achieved / partially achieved / not achieved**
  * ♡ tokens are awarded for time contributed regardless of outcome (you showed up and worked)
  * Outcome result is permanently attached to the record for accountability and reputation
* All content and results are exportable and attached to the Project ID

#### Method 3 — Team Collaboration (3-Person Minimum, Role-Required)
* A team of **3 or more** works on a project together
* **Required role coverage** (minimum 1 of each, can have more members):
  * **Technology Representative** — technical execution and feasibility
  * **Creative Representative** — design, UX, ideation, content
  * **Business / Value Representative** — strategy, ROI, market fit, stakeholder value
* **Before starting:**

## 8.5) API-First Platform & SDK (Embeddable Integration)

### 8.5.1 Architecture Principle

The entire SoI Governance Engine is **API-first**. Every feature accessible in the web UI must also be accessible via a public REST API. This enables external companies to run the governance engine inside their own code, websites, or products — without ever visiting the SoI web app.

### 8.5.2 SDK (Client Libraries)

* Provide official SDKs (JavaScript/TypeScript first, Python second) for embedding
* SDK wraps all REST API endpoints with typed methods
* SDK handles: authentication (API key + OAuth), session management, webhook callbacks, error handling
* Embeddable UI components (Web Components / iframe-based) for companies that want drop-in UI
* Headless mode: companies can use the API/SDK with their own custom UI

### 8.5.3 Scoping Hierarchy: Project → Differentiator → Specification

Every SDK/API interaction is scoped to a three-level hierarchy:

* **Project (Project ID):** Top-level container representing a company's product, initiative, or system. Each project has its own configuration, sessions, token rules, and data isolation.
* **Differentiator (Differentiator ID):** Within a project, differentiators represent distinct dimensions, features, or hypotheses being evaluated. Example: "UX Approach A vs B", "Pricing Model X", "Feature Priority Set".
* **Specification (Specification ID):** Within a differentiator, specifications are the concrete parameters, constraints, or configurations to fine-tune and simulate. Example: target audience, pricing tiers, feature toggles, governance rules.

This hierarchy allows companies to:
* Run multiple polling/governance sessions scoped to specific projects
* A/B test ideas via differentiators
* Fine-tune and simulate existing products or new concepts via specifications
* Compare outcomes across differentiators and specifications within a project

### 8.5.4 API Key & Tenant Management

* API keys issued per organization (tenant)
* Each API key scoped to one or more projects
* Rate limiting per API key (separate from web UI rate limits)
* Usage metering for billing (API calls, sessions, AI processing)
* Webhook registration for async events (themes ready, ranking complete, etc.)

### 8.5.5 Embeddable Modes

| Mode | Description |
|------|-------------|
| **Full embed** | Drop-in UI component (iframe / Web Component) — full session flow rendered inside host app |
| **Headless API** | No UI — company calls API/SDK endpoints and renders their own UI |
| **Hybrid** | Company uses some embedded components (e.g., ranking widget) and custom UI for others |

### 8.5.6 Core Entities (additions for SDK scoping)

**Project**
* id, org_id, name, description, created_at, config (token rules, AI provider, language defaults)

**Differentiator**
* id, project_id, name, description, hypothesis, created_at

**Specification**
* id, differentiator_id, parameters (JSON), constraints, version, created_at

**Session** (updated)
* Now includes: project_id, differentiator_id, specification_id (all optional for standalone web use; required for SDK/API use)

### Monetization (MVP1 — Stripe)

#### Moderator Fee (Required)
- **Moderator must pay** to use the tool (per-session fee via Stripe)
- **Cost splitting option:** Moderator can choose to split the fee equally among participants
  * Example: $100 fee / 100 users = $1 per person
  * Example: $100 fee / 200 users = $0.50 per person
  * Fee per user is calculated dynamically as participants join (displayed in real-time)
  * If Moderator does not enable cost splitting, Moderator pays the full fee
- **Paying members receive Polling Results** — any user who pays their share automatically receives the full session output

#### Gamified Contribution Reward
- **Moderator-funded bonus:** Moderator can set a reward amount (e.g., $25) to be awarded to one participant
- **Appears random to users** but is determined by the **Contribution Quality Score (CQS)** — a hidden rating system
- **CQS scoring is selective — NOT applied to all responses:**
  1. Cube 7 determines the #1 most-voted Theme2 cluster
  2. Only responses assigned to that cluster with **>95% theme confidence** are eligible
  3. CQS Composite Score is calculated ONLY for eligible responses (saves API calls — no wasted scoring on ineligible responses)
  4. Highest CQS wins. If tie, **randomize among tied responses** (fair and unpredictable)
- **CQS Rating Metrics and Composite:** See **Cube 6 — AI Theming Clusterer** section for the full 6-metric definition (Insight, Depth, Future Impact, Originality, Actionability, Relevance) and composite weights. Moderator-adjustable per session.
- **Visibility:** CQS scores hidden from all users. Moderators see scores in their dashboard. System uses scores for reward selection and analytics.
- **Tie-breaking:** If CQS tie, **randomize** among tied responses (fair and unpredictable)

#### Access Rules
- **Lead/Developer exception:** Leads ALWAYS get free access to results (transparency + accountability)
- **Payment providers at launch:** Stripe, Google Pay, Apple Pay

## 13) Production Architecture Constraints

### Architectural Rules
* **No row-by-row API calls** — batch all embedding requests efficiently
* **No disk-heavy CSV intermediates** — stream data through pipeline in memory
* **No repeated full clustering** — use streaming/incremental MiniBatchKMeans updates
* **Horizontal scaling** — all services must be stateless or use shared state (Redis/Postgres)
* **Rate limiting** on all public endpoints
* **Anti-sybil safeguards** on voting and response submission
* **Governance weight damping** to prevent manipulation
* **Circuit breaker pattern** for all external AI provider calls (OpenAI, Grok, Gemini)
* **Graceful degradation** — system must serve partial results if AI pipeline is delayed

### Determinism Requirements
All clustering and ranking operations must be fully reproducible:
* **Seeded MiniBatchKMeans** — fixed `random_state` for identical outputs
* **Fixed embedding model version** — pin exact model ID, never auto-upgrade
* **Stable cluster ordering** — deterministic sort after clustering
* **Replay reproducibility hash** — SHA-256 of (inputs + parameters) = expected output hash
* **Version-locked cube dependency graph** — every cube pins its upstream/downstream versions

### Background Processing Architecture
* **Task queue:** Celery or async worker pool for background AI processing
* **Embedding worker fleet:** Dedicated batch workers, horizontally scaled
* **Streaming clusterer:** MiniBatchKMeans with deterministic seed, incremental updates
* **Governance engine:** Voting weight compression, quadratic normalization
* **Append-only token ledger:** Immutable entries with lifecycle states

### Failure Mode Handling
* **AI provider outage:** Circuit breaker → queue → exponential backoff → failover to next provider → cached embedding fallback
* **Embedding backlog surge:** Back-pressure → sampling mode → priority queue
* **Partial cluster update:** Atomic batch commits → rollback on failure → serve last stable cluster
* **Redis failure:** Graceful degradation to Postgres → reconnect with backoff
* **Postgres failover:** Read replicas → automatic promotion → connection pool retry
* **Burst queue overflow:** Load shedding via sampling → 503 + retry-after header
* **Governance manipulation:** Anti-sybil detection → weight damping → anomaly flagging → audit log

## 14) Governance Compression & Token Math

See `Token_Governance_Math.md` for full formal specification. Summary:

### Governance Requirements
* **Governance weight damping** — prevent any single actor from dominating outcomes
* **Quadratic vote normalization** — diminishing returns on repeated votes: `weight = sqrt(tokens_staked)`
* **Token velocity caps** — limit transfer/redemption speed to prevent gaming
* **Reputation multipliers** — earned trust increases governance weight over time
* **Drift detection** — `cosine(previous_centroid, new_centroid)` to detect theme drift between runs
* **Cluster stability scoring** — measure consistency of theme assignments across re-runs

### Treasury Accounting
* **Treasury-backed HI redemption** — HI tokens only redeemable against available treasury balance
* **Token dispute workflow** — flag → review → resolve with complete audit trail
* **Version-locked ledger** — every entry references cube version + dependency graph hash
